# CORS Preflight Handling at Container Apps Ingress vs. Application Level

!!! info "Status: Planned"

## 1. Question

When a browser sends a CORS preflight request (HTTP OPTIONS with `Origin` and `Access-Control-Request-*` headers), is the preflight handled at the Container Apps ingress (Envoy) level or forwarded to the application? And when the application's CORS configuration is incomplete (e.g., missing `OPTIONS` route), does the preflight fail with a 405 or 404, causing CORS errors for all cross-origin requests?

## 2. Why this matters

CORS misconfigurations are a frequent source of support tickets. On Container Apps, unlike some other platforms that handle CORS at the gateway level, CORS responses must be generated by the application (the ingress does not automatically add CORS headers). When the application framework handles CORS via middleware (Flask-CORS, Django-cors-headers, Express CORS middleware), OPTIONS preflights are usually handled correctly. When teams manually implement CORS routes without handling OPTIONS, or when the framework is misconfigured, OPTIONS returns 404 or 405, blocking all cross-origin API calls.

## 3. Customer symptom

"API calls work in Postman but fail from the browser with 'CORS error'" or "OPTIONS requests return 404" or "We added CORS headers to GET but POST still fails — the preflight is rejected."

## 4. Hypothesis

- H1: Container Apps ingress (Envoy) does not perform CORS preflight handling by default — it forwards OPTIONS requests to the application like any other HTTP request. The application is solely responsible for responding to OPTIONS with the correct `Access-Control-Allow-*` headers.
- H2: When the application does not have a route for OPTIONS (common in Flask without CORS middleware), OPTIONS returns 405 Method Not Allowed. The browser interprets a non-2xx preflight response as a CORS failure, blocking the actual cross-origin request.
- H3: Adding CORS middleware (e.g., `flask-cors`) adds an OPTIONS handler automatically and injects the correct `Access-Control-Allow-*` headers, resolving the preflight failure without manual route definition.
- H4: The `Access-Control-Allow-Origin: *` header on POST/GET responses is insufficient alone — the preflight OPTIONS response must also include `Access-Control-Allow-Methods` and `Access-Control-Allow-Headers` matching the actual request.

## 15. Reproduction notes

- "Simple" cross-origin requests (GET, POST with `application/x-www-form-urlencoded` or `text/plain`) do not trigger preflights. Requests with `Content-Type: application/json` or custom headers trigger a preflight OPTIONS.
- Container Apps ingress does NOT handle CORS — it is entirely an application responsibility.
- `flask-cors` library: `pip install flask-cors`; `from flask_cors import CORS; CORS(app, origins="*")`.
